# 可测试性评估报告

> 评估版本: V1
> 日期: 2026-05-25
> 评估方法: 从 20 个跨域集成测试场景（S01-S20）倒推，结合 4 份技术架构方案分析
> 涉及文件: 权限中心、调度引擎、审计日志、部署框架技术方案

---

## 1. 评估概要

| 维度 | 等级 | 关键瓶颈 |
|------|------|---------|
| 环境可测试性 | ⚠️ 中 | GPU 资源依赖度高，多租户环境搭建复杂 |

## 2. 按维度的详细评估

### 2.1 环境可测试性

#### 问题 1: GPU 资源是硬约束（影响 S01, S06-S08, S16-S17, S20）

**跨域场景覆盖分析**:

| 场景 | GPU 需求 | 预估并行度限制 |
|------|---------|--------------|
| S01 全链路 | 4 卡 DDP | 同一时间最多 1 组执行 |
| S06 调度告警 | 10+ 节点 | 需要独立 GPU 集群 |
| S07 节点维护 | 多节点 | 需要 >2 节点集群 |
| S08 抢占保护 | 多节点 | 需要 >2 节点集群 |
| S16 Jupyter + DDP | 4 节点 | 独占资源 |
| S20 弹性伸缩 | Min=0 → N | 需要 HPA 真实生效 |

**评估**: GPU 集群成本高、并行度低，20 个场景中有 16 个需要真实 GPU 硬件。全部串行执行约 6-8 小时，但在共享集群上因资源争抢可能延长到 12h+。

**建议**:
1. 搭建两套测试环境: 小型 GPU 集群（2 节点 × 4 卡）用于 P0 冒烟 + 开发自测；完整集群用于 P1 全量回归 (夜间)
2. 调度器核心逻辑（排队、策略选择）可用 Fake K8s 客户端在单元测试覆盖，无需真实 GPU
3. 弹性伸缩（S20）可在非 GPU 服务上验证 HPA 行为，推理冷启动部分单独测试

#### 问题 2: 多租户环境搭建成本高（影响 S02, S03, S14）

场景 S02 要求 3 个租户、每个 2+ 团队；S03 要求 4 个预置角色。每个租户需要独立的飞书应用配置。

**评估**: 手动准备 3 套飞书 OAuth 凭证繁琐，且测试数据相互依赖导致清理复杂。

**建议**:
1. 提供 `make seed-test-env` 命令: 一键创建 3 租户 + 6 团队 + 4 角色 + 预置测试用户
2. 提供 `make cleanup-test-env` 命令: 级联清理所有测试数据
3. 飞书 OAuth 在测试环境提供一键绑定/解绑的能力

### 2.2 时间敏感行为（最大可测试性风险）

#### 问题 3: 大量硬编码时间窗口无法加速（影响 S07-S11, S20）

| 场景 | 时间参数 | 正常耗时 | 对自动化的影响 |
|------|---------|---------|--------------|
| S07 节点排空 | 30s 心跳超时 + 30s 优雅终止 | ~60s | 可接受但累积后拉长 |
| S08 抢占保护 | 5min 最小保证期 + 24h 3 次限制 | >25min | 🔴 阻塞自动化 |
| S09 故障漂移 | 30s 确认窗口 + RTO < 5min | ~6min | ⚠️ 耗时较长 |
| S10 熔断恢复 | 30s Half-Open + 10s 错误率窗口 | ~40s | 可接受 |
| S11 灰度 | 每阶段 5min × 3 + 自动回滚 | ~20min | 🔴 阻塞自动化 |
| S20 毛刺防护 | < 30s 不扩容 | ~30s | 可接受 |

**评估**: S08（抢占保护 5min）和 S11（灰度分阶段 5min）是全自动化测试的最大障碍。如果真实等待，2 个场景就需要约 45 分钟。

**建议**:
1. **时间加速配置**: 将硬编码常量抽取为环境变量/配置项，测试环境设短值:
   - `PREEMPT_MIN_GUARANTEE_SECONDS` → 正常 300，测试设 5
   - `PREEMPT_DAILY_LIMIT_WINDOW_HOURS` → 正常 24，测试设 0.01
   - `GRAY_CANARY_PHASE_MINUTES` → 正常 5，测试设 0.1
   - `HEARTBEAT_TIMEOUT_SECONDS` → 正常 30，测试设 3
   - `FAULT_CONFIRMATION_WINDOW_SECONDS` → 正常 30，测试设 3
2. **通过时间点标记避免真实等待**: 对"24h 内最多 3 次"这类日历窗口，提供 `POST /internal/test/fast-forward` 接口，允许测试推进时间而不需要真的 sleep
3. **灰度测试分离**: 灰度发布逻辑（S11）拆分为"策略引擎"（纯逻辑，毫秒级）和"执行引擎"（需真实部署），前者可完全脱离时间约束测试

#### 问题 4: JWT 短时效 + 缓存 TTL 影响测试确定性（影响 S03, S14）

| 参数 | 值 | 问题 |
|------|----|------|
| JWT expiry | 15min | 测试中 Token 可能在用例执行中途过期 |
| 权限缓存 TTL | 14min | 修改角色后，最多 14min 才能生效 |
| 黑名单 TTL | 随 Token 剩余时间 | 吊销后立即生效？还是等到缓存过期？ |

**评估**: S03 中"权限变更后刷新页面 → 菜单立即更新"这个预期在架构上不完全成立。14min 的缓存意味着权限变更有延迟生效窗口。

**建议**:
1. 提供 `POST /api/v1/internal/test/clear-permission-cache?user_id=xxx` 测试接口，绕过 TTL 直接清缓存
2. 在跨域测试场景 S03 的预期结果中注明"最大生效延迟 = 缓存 TTL"而非"立即生效"
3. JWT 在测试环境签发 TTL 设为 1h，避免单场景执行中途过期

### 2.3 异步与最终一致性

#### 问题 5: 审计日志异步写入延迟（影响 S01, S14, S18）

审计事件经 Redis Streams → Audit Worker → PG，架构上端到端延迟 < 500ms，但存在以下可测试性问题:

| 问题 | 影响 | 概率 |
|------|------|------|
| Redis Streams 积压 | 延迟从 500ms 扩展到数秒 | 低 |
| Worker Crash 后重启 | 未 ACK 事件重放，出现重复 | 低 |
| 死信队列事件 | 事件未写入但无报错 | 极低 |

**评估**: S14 要求验证"每操作都有审计日志"，测试中需要在业务操作后轮询等待审计记录出现。如果 Worker 延迟不稳定，轮询超时时间需要保守设置（5-10s）。

**建议**:
1. 审计日志查询 API 增加 `wait_for_event=true` 参数: 服务端等待最多 3s，确保事件已落库再返回
2. 提供审计健康检查端点: `GET /api/v1/internal/audit-health`，返回当前消费延迟和积压数
3. S14 的"审计完整性校验"场景中，在业务操作后使用 `wait_for_event` 而不是固定 sleep

#### 问题 6: 计费最终一致性（影响 S18）

费用计算在任务完成后触发，涉及定价配置查询、折扣计算、配额更新三个步骤，且日汇总和月结算是延迟的。

**评估**: S18 要求"费用流水与审计日志交叉验证"，但计费和审计是两个独立异步管道，时间上存在偏差窗口。

**建议**:
1. 计费结果提供 `status` 字段（pending/settled），场景 S18 的断言应等 `status=settled` 后再验证
2. 交叉验证工具化: 提供 `GET /api/v1/internal/cross-verify?task_id=xxx`，返回计费和审计两边数据的一致性报告

### 2.4 外部依赖模拟

#### 问题 7: 飞书 OAuth 无法在 CI 中真实调用（影响 S01-S03）

**评估**: 飞书 OAuth 需要浏览器交互和飞书客户端，自动化测试无法走真实流程。

**建议**:
1. 内测/CI 环境提供 bypass 登录: 通过预设 Service Token 直接换取 JWT，无需飞书 OAuth
2. 飞书消息通知 Mock: 提供 `POST /api/v1/internal/test/mock-feishu-notification`，验证通知内容而不真的发送
3. 集成测试使用飞书 Sandbox 应用（测试版飞书应用），不影响正式用户

#### 问题 8: K8s/Volcano 操作的真实性边界（影响 S01, S04, S06-S08）

**评估**: 调度器的高度依赖 K8s 和 Volcano CRD，完全 Mock 失去测试意义，完全真实又太慢。

**建议**:
采用"分层测试"策略:

| 层 | 范围 | 实现方式 | 执行时间 |
|----|------|---------|---------|
| 单元测试 | 调度策略算法、排序、配额校验 | Go mock/fake | 毫秒级 |
| 集成测试 | API 层 + 数据库 + Redis | 真实 PG + Redis，Mock K8s | 秒级 |
| E2E 测试 | 跨域场景 | 小型 Kind 集群 + 真实 K8s | 分钟级 |

- 调度策略选择（FIFO/Fair/Priority/Hybrid）可纯单元测试
- GPU 拓扑感知调度可 Mock NVLink topology 数据
- Volcano CRD 操作在 E2E 层覆盖，单元/集成层用 Fake K8s Client

---

### 2.5 测试数据管理

#### 问题 9: 数据集版本引用链难以构造（影响 S01, S19）

**评估**: S19 涉及数据集 v1 → v2 → 正在运行的任务引用 → 删除校验 → 权限变更的复杂状态组合，手动构造数据繁琐。

**建议**:
1. 提供 `make seed-dataset-chain` 命令，一键创建含 v1/v2/引用/权限变更的完整数据集链
2. 数据集服务暴露内部 API: `POST /internal/test/datasets/:id/force-status`，允许将数据集设为"被引用"状态以测试删除拦截

#### 问题 10: Checkpoint 数据的模拟（影响 S07）

**评估**: 断点续训验证需要真实的训练 checkpoint 文件（数百 MB），且在 GPU 上运行真实训练步骤。

**建议**:
1. 提供最小化测试用 checkpoint（一个小网络 + 几步训练），可在 CPU 上快速验证
2. 续训逻辑（checkpoint 加载 + loss 曲线衔接）可在单元测试中通过 Mock Checkpoint Manager 覆盖
3. E2E 测试仅验证"保存 checkpoint → 重启 → 加载"流程是否跑通，不校验 loss 曲线精确衔接

## 5. 按执行顺序的建议落地步骤

```
Phase 1（开发初期）:
  ├── R5: bypass 登录 → 让所有测试不需要飞书
  ├── R1: 时间常量配置化 → 消除最大阻塞因素
  └── R6: seed/cleanup 命令 → 环境准备自动化

Phase 2（开发中期）:
  ├── R2: 时间推进接口 → S08/S11 可自动化
  ├── R3: 清缓存接口 → S03 权限测试确定性
  ├── R4: wait_for_event → 审计相关场景稳定
  └── T5: CI Kind 集群 → E2E 持续集成

Phase 3（集成测试前）:
  ├── R7: 数据集 seed → S01/S19 数据准备
  ├── T1-T2: 分层测试设计 → 执行效率提升
  ├── T3: Chaos Mesh 引入 → 故障类场景
  └── T6: 夜间回归 GPU 场景 → 资源利用率优化
```

## 6. 最终评估总结

**总体可测试性评分: 6.5/10**

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | 8/10 | 微服务拆分清晰，内部 API 契约明确 |
| 时间可控性 | 4/10 | 🔴 最大短板，需上市前完成 R1-R2 |
| 环境准备 | 5/10 | GPU 成本高，但 seed 脚本可缓解 |
| 外部依赖 Mock | 6/10 | 飞书 OAuth 有 bypass 方案，K8s 可分层测试 |
| 数据隔离 | 9/10 | 行级隔离优秀，支持并行测试 |
| 幂等与回滚 | 9/10 | Helm + goose 方案完备 |
| 可观测性 | 7/10 | 标准化健康检查和指标，但需要补充测试专用观察点 |

**最大风险项**: 时间相关测试场景（S08, S11）如果不能通过配置化缩短等待时间，将导致自动化测试执行时间过长，最终被迫跳过关键验证。

**关键结论**: 只要在开发阶段落地 R1（时间常量配置化）、R5（bypass 登录）和 R6（seed/cleanup 命令），20 个跨域场景中 18 个可实现全自动化回归。剩余 S09（故障漂移）和 S15（多集群灾备）因环境限制需保留手动演练。
