### S03 权限控制对全流程的约束（正向+负向全覆盖）

**涉及故事**: US1.1.2 → US1.1.3 → US1.1.5 → US3.1.5 → US3.2.4 → US4.3.1 → US4.2.7

**前置条件**: 预置 admin/ops/dev/readonly 四个角色，各有明确权限边界

| # | 步骤 | 预期结果 |
|---|------|---------|
| 1 | admin 用户完成"数据→训练→部署"全流程 | 全流程成功，无权限阻碍 |
| 2 | dev 用户执行上述全流程 | 同样成功（dev 应有开发和部署权限） |
| 3 | readonly 用户打开功能页面 | 菜单可见但操作按钮 disabled |
| 4 | readonly 用户直接调用部署 API（US1.1.5） | 返回 403，不执行业务逻辑 |
| 5 | readonly 用户尝试下载模型文件（US3.2.4） | 返回 403 |
| 6 | readonly 用户尝试创建告警规则（US2.1.4） | 操作按钮不可用或 API 返回 403 |
| 7 | 自定义角色：只有"训练"权限无"数据集"权限的用户提交训练 | 无法选择数据集，任务创建失败 |
| 8 | 数据集设为"团队内"（US3.1.5），非团队成员通过 API 访问 | 返回 403 |
| 9 | 数据集权限从"公开"改"私有"后，非授权用户之前引用的任务 | 新训练提交时校验失败 |

**关键断言**:
- 每一个跨域操作点，权限校验是"必经之路"
- 权限变更后，刷新页面/JWT 续期后效果应同步
- API 层的 403 返回不应暴露内部实现细节
